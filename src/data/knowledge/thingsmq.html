一、物联网系统基础架构与核心挑战

  • 物联网系统基础架构由设备接入、协议接入、数据处理、应用四个核心层组成，与移动通信网络（终端→RU→DU→核心网）相对应。
  • 三大核心挑战：
    • 设备接入协议种类多：物联终端协议百花齐放，定制开发工作量大，需考虑不同协议的会话管理和编解码。
    • 设备对接工作任务繁：持续性工作，团队难以专注业务开发，难以形成统一对接规范。
    • 多源异构数据处理难：终端上报数据格式难以统一，海量数据接入对系统提出更高要求。

二、物联网设备接入协议

  • 物联网协议体系按网络分层：应用层 / 传输层 / 网络层 / 数据链路层 / 物理层。
  • 常见应用层协议： MQTTHTTP(s)
  • 行业专有协议： HJ212（环保）JT/T808（交通）SL651（水利）Modbus（工业）OPC UA（工业）IEC104（电力）BACnet（建筑）GA1400（安防）KNX（楼宇）
  • 关键概念：标准协议定义了报文格式和交互流程，但协议承载的业务内容字段（Data）没有统一标准，各平台规范互不通用。
  • 传输层核心：TCP / UDP；网络层：IPv4 / IPv6。

三、ThingsMQ产品定位与架构

  • ThingsMQ 是中国电信天翼物联推出的轻量级、可扩展的物联网消息队列中间件。
  • 面向中小型物联网项目，提供多协议互联互通、数据桥接集成能力。

两种部署形态

  • 边缘网关部署：网关+组件一体化交付，开箱即用；适用本地数据采集；支持多种CPU架构。
  • 云服务部署：按需资源分配，支持公有云/私有云/混合云；分钟级K8S云原生部署。

逻辑架构

  • 接入服务（协议插件）→ 规则引擎（解析/转换/流转）→ 桥接服务（数据库/消息队列）。
  • 集成架构：支持与数据库/运维监控中间件无缝集成，可基于 Kubernetes 统一托管。

四、插件管理机制

  • 按来源分：内置插件——开箱即用，覆盖主流标准协议和常见桥接目标；定制插件——按项目需求开发，满足私有协议接入或特殊桥接场景。
  • 按功能分：协议插件——负责设备接入和协议编解码；桥接插件——负责数据向下游系统推送。
  • 动态加载/热插拔：无需服务中断，通过动态模块化架构确保高可用性和业务连续性。

五、规则引擎（数据解析转换）

  • 通过脚本语言编写 + 流程编排构建自动化规则引擎，实现数据映射、过滤、计算、标准化输出。

SQL数据处理单元

  • 输入数据记作 DEVICE_DATA 虚拟表。
  • 支持字段提取、重命名、添加、条件过滤。
  • 输出仅含 SELECT 中列出的字段；不满足 WHERE 条件的数据被过滤。

JavaScript数据处理单元

  • 三个内置对象：input（接收数据）、output（输出结果）、error（调用 error.failed() 丢弃数据并记录原因）。
  • 支持字段提取、数值转换、异常处理。

六、二次开发三种方式

  • ① 数据解析转换（核心自带能力）：开发SQL/JS脚本，实现数据接入、转换、过滤。
  • ② 协议插件开发：利用协议加载框架，开发私有协议接入插件，扩展设备和数据源接入能力。
  • ③ 桥接插件开发：利用桥接框架，开发桥接插件，扩展应用集成和数据目的地推送能力。

七、协议插件开发

技术栈

  • Java 21 + Spring Boot 3.3.5（必须与组件一致）

必须实现的接口

  • PluginDaStartService：插件初始化（init）与卸载（destroy），设备接入层初始化。
  • DeviceAutoOffLineService：设备断连回调（autoOffLine），清理设备状态。

可选接口

  • ProtocolReceiveCmdService：设备指令下发。

PluginInitContext上下文字段

  • applicationContext（Spring容器）
  • pluginAttributes（配置参数Map）
  • pluginName（插件名称）

设备协议处理7大环节

  1. 设备连接处理（必选）
  2. 心跳保活（视协议）
  3. 数据上报（必选）
  4. 指令下发（可选）
  5. 设备生命周期（必选）
  6. 协议编解码（必选）
  7. 插件生命周期（框架要求）

三种实现方式

  • 方式一：Netty监听+协议编解码：适用自定义TCP/UDP/HTTP协议；灵活度最高；开发量大。
  • 方式二：开源协议栈+框架适配：适用MQTT/CoAP/Modbus等标准协议；开发量适中；灵活度受限。
    • MQTT→Moquette/EMQX
    • CoAP→Californium
    • Modbus→Modbus4J
    • JT/T808→开源808框架
  • 方式三：集成SDK+框架适配：适用云平台桥接/厂商SDK对接；开发量最小；灵活度最低。

八、桥接插件开发

  • 技术栈与协议插件一致（Java 21 + Spring Boot 3.3.5）。
  • 必须实现 PluginBridgeService 接口：init(PluginInitContext) 建立外部系统连接、destroy() 关闭连接释放资源。
  • 需编写 jsonSchema 配置文件定义桥接目标参数（地址、端口、认证等）。
  • destroy() 中必须释放外部连接，防止资源泄漏。
  • 内置HTTP桥接插件支持推送设备上线、数据上报、设备下线事件到下游系统。

九、二开环境与典型场景

环境准备

  • 必备：Java JDK 21（配置 JAVA_HOME）、浏览器（Chrome / Edge）
  • 选配：MQTTX（MQTT调试）、Apifox（API调试）、TCP工具、数据库（MySQL）、消息队列（Kafka/Pulsar/RocketMQ）、Docker、AI辅助工具

典型应用场景

  • 智慧园区：设备接入→数据解析→消息流转，实现一张图管理。
  • 低空经济：边缘网关+ThingsMQ软硬一体，协议归一统一上云。
  • 跨网解决方案：通过桥接插件适配跨网交换系统，解决跨网域数据传输难题。
  • 1+X赋能：应急管理、民生工程、工业数采、市政监管、乡村振兴等领域。
