1.1 CV 二开定义与背景 ★高频

CV 二开（计算机视觉二次开发）：为满足各省市 CV 监控中长尾 AI 场景需求，加速二开交付效率、降低成本；由视联网 & AI 公司打通算法标训平台、推理平台与视联网应用，对省市二开需求进行支持。

核心思想：如何利用视联各平台底座能力，快速搭建二开应用。

1.8 算力成本公式 ★高频▲难点

每路算力成本 = (硬件投入设备数 × 硬件成本) / 总路数
            = (算法路数 / 算法QPS / 抽帧频率) × 硬件成本 / 总路数
            ≈ 硬件成本 / 总路数   （当 路数 < QPS × 抽帧频率）

核心原则（必背金句）：

无论一路加载多少算法，抽帧永远按照加载算法的最小抽帧频率进行一次抽帧即可。
  • 省节点资源一般冗余 20% 作为安全区，POC 测试在冗余资源上。
  • 结算链路：云公司 → 出账 → 天翼视联 → 出账 → 省公司 → 结算 → 地市公司。

明厨亮灶案例（A10，元/路月）：抽帧 0.94；老鼠/拖把清洗灶台 2；抽烟 0.52；画面异常/地面损坏 0.0003；其余（厨师服/帽/口罩/玩手机/垃圾桶/手套/挡鼠板）0.052；合计 5.8246 元/路月。

3.2 ThingsMQ 定位、形态与性能

ThingsMQ：中国电信天翼物联的轻量级、高性能、可扩展物联网消息队列中间件，面向中小型物联网项目，提供海量设备连接、低延时消息传递、多协议互联互通、数据桥接集成。

两种部署形态：

形态特点
边缘网关部署网关 + 组件一体化，开箱即用，多 CPU 架构
云服务部署公有/私有/混合云，分钟级 K8S

性能指标：单节点十万并发 / 集群百万并发；消息传输延迟低至毫秒级；协议插件覆盖 90%+ 物联接入场景；10+ 数据桥接组件。

金句："ThingsMQ 中间件简化了设备接入流程，实现了设备层和业务层的解耦，应用更专注于业务。"

3.3 ThingsMQ 三大核心组件 ★高频

组件作用支持
物联协议插件槽（device access）设备接入 + 协议编解码MQTT/LwM2M/CoAP/Modbus/HJ212/OPCUA/IEC104/JTT808/BACnet/HTTP/私有
规则引擎（rule）数据解析 → 格式转换 → 数据过滤 → 数据流转SQL + JS 脚本
数据桥接器（bridge）数据向下游系统推送TDengine/Kafka/MySQL/Tidb/Pulsar/RocketMQ/AMQP/Redis/HTTP/MQ

运维：监控中心 + 节点管理（K8S 统一托管）。

3.4 ThingsMQ 三种二开方式 ★高频

方式说明
① 数据解析转换直接用核心自带能力，开发 SQL/JS 脚本
② 协议插件开发扩展设备和数据源接入
③ 桥接插件开发扩展应用集成和数据目的地推送
二开 = ① and/or ② and/or ③。

插件开发通用 5 步：创建 Spring Boot 项目 → 实现框架接口 → 编写 jsonSchema 配置 → 打包可执行 jar → 导入组件配置启停。

协议栈开源选型：MQTT→Moquette/EMQX；CoAP→Californium；Modbus→Modbus4J；JT/T808→开源 808 框架；阿里云 IoT→AMQP；华为云 IoT→AMQP/MQTT；OneNET→HTTP 推送回调。

第五章 量子密信 SDK 客户端二开

覆盖课件：【燎原计划】基于量子密信 SDK 的客户端二开

5.1 产品定位 ★高频

量子密信 SDK：中电信量子推出的企业级加密即时通讯 SDK。

  • 多平台：Android、iOS、HarmonyOS、Windows
  • 多形态产物：核心 SDK、Lib SDK、Kit SDK，统一接口、规范错误码
  • 符合国密标准，基于量子密钥，通过安全介质 + 安全中间件实现：
  • 消息一次一密
  • 音视频全帧加密
  • 支持文字、语音、图片、视频、文件；加密单聊/群聊、加密企业通讯录

5.2 解决的四大问题

  1. 现有 APP 无法嵌入密信 APP（系统限制，需要可接入 SDK）
  2. 基于密信 APP 二开成本高（公版业务庞杂，功能与界面高度耦合）
  3. 客户只需部分能力或底层 IM/RTC，UI 完全自定义 → 需要能力解耦
  4. 传统 IM/RTC 安全性不足，政企金融缺乏原生量子加解密

5.3 三种 SDK 形态 ★高频

维度核心版 (Core)Lib 版Kit 版
内容核心 IM/RTC/量子加解密仅 API 接口与核心功能（底层）预置界面组件 + 业务逻辑
UI无开发者自定义官方 UI 组件，深度定制
复杂度最低高（需自主渲染消息列表/UI）低（简化 80% 编码）
适用基础能力功能驱动、深度定制 UI/UX界面驱动、快速上线
UI Kit 与 Lib 可组合使用：Lib 实现核心功能，选择性集成 UI Kit 非关键模块，平衡效率与灵活性。

5.4 两种集成模式 ★高频

模式SDK + 公版 IM 服务SDK + 私版 IM 服务
部署公有化部署全量私有化部署，数据不出局
互联互通与公版密信互联互通（同账号、消息互通）无需与公版互通
适用省分公司/专业公司集成；已有 APP 引入能力有自有办公平台底座、有安全诉求
案例安徽量子云总机、北京"易办"、安徽 IWORK湖南 OA / 湘办通

5.5 技术架构 ▲难点

  • 应用与应用服务器交互业务信息（账号体系、用户数据），密信 SDK 不维护开发者应用数据。
  • 应用服务器与密信 SDK 服务器做身份验证，返回信息存入开发者应用服务器数据库。
  • 消息发送经密信 SDK 服务器转发给其他终端。

登录及消息发送流程：用户登录 App 获取 Token → 登录密信 SDK → IM 服务连接成功 → 可发送消息。

RTC 架构分层（自下而上）：基础能力层 → RTC SDK（业务逻辑层）→ RTC Kit（UI 交互层）→ APP 层。

UI Kit 架构（Android）：采用 MVVM（Model–View–ViewModel），UI 展示与业务逻辑独立；回调数据由 RecyclerViewAdapter 按 type 判定 UI 样式。

5.6 量子加解密（量子安全盾）▲难点

基于国密标准，提供：身份认证、数字证书、密码运算、密钥管理。

  • 安全介质：量子安全 U 盾、量子 SIM 卡、量子 TF 卡。
  • 密钥全生命周期：充注、申请、查看、销毁、分发、协商、调度、供给。
  • 密码服务节点：KMS、密码机、QKD/KM。
  • 服务：身份认证、单点登录、SSL VPN、CA 证书、签名验签、数字信封、消息摘要。

第六章 TOGAF 企业架构设计与实践

覆盖课件：燎原计划-大型企业架构设计与实践(TOGAF)（最大文件，本章为全局重点）

6.1 企业架构定义与意义 ★高频

企业架构是业务流程和 IT 的组织逻辑，反映了企业运营模型的集成和标准化需求，作为连接战略和 IT 的桥梁，将业务战略愿景转换为高效率运营的规则、过程和实施路线图，是构建企业核心竞争力之本。
  • 世界 500 强中超 80% 跨国企业使用企业架构（微软、IBM、华为）。
  • TOGAF 由欧洲 The Open Group 于 1995 年开发出第一个版本。

建设意义（5 点）：①推进业务与数字技术融合创新 ②支持数字化转型战略 ③优化业务流程 ④提升组织管理效能 ⑤推进企业发展变革。

6.2 "4A"架构 ★高频

架构缩写英文定位
业务架构BABusiness Architecture业务结构化描述：运营模式/流程体系/组织结构/地域分布
数据架构DAData/Information Architecture数据结构化流转：数据流蓝图
应用架构AAApplication Architecture应用结构化表达：支持业务并处理数据的应用功能
技术架构TATechnology ArchitectureIT 基础设施：软硬件组件
安全架构常作为扩展架构。
业务架构是源头；数据、应用、技术架构承接业务架构，把现实世界"孪生"到数字世界。
关系：BA → 输入/支撑 → DA/AA → 实现 → TA。

6.3 TOGAF ADM 架构开发方法 ★高频▲难点（核心中的核心）

ADM 定义：一个用于开发架构的、经测试并可重复的流程。所有活动在一个连续的架构定义与实现迭代周期内实施，使组织能以受控方式实施企业转型。

ADM 各阶段（必背顺序与目标）：

阶段名称核心目标
预备阶段Preliminary确定并建立组织期望的架构能力（EA 能力）；按需定制 TOGAF、定义架构原则
阶段 A架构愿景识别关键利益相关者，形成架构愿景文档；获得架构工作声明批准
阶段 B业务架构开发目标业务架构，响应战略驱动因素；差距分析识别备选路线图
阶段 C信息系统架构（数据 + 应用）开发目标信息架构，含数据架构与应用架构
阶段 D技术架构开发目标技术架构，通过技术组件/服务交付架构愿景
阶段 E机会和解决方案基于 B/C/D 差距分析生成架构路线图初始完整版本；基于 ABB 定义整体 SBB；识别过渡架构
阶段 F迁移规划最终确定详细实施与迁移计划；解决"如何从基线架构向目标架构转移"
阶段 G实施治理通过实施项目确保与目标架构一致；与敏捷软件开发对齐
阶段 H架构变更管理确保架构生命周期维持、治理框架执行、EA 能力满足需求
需求管理Requirements Management贯穿整个 ADM 持续运行，是架构开发的中心

ADM 特性：是逻辑方法，不应被视为顺序流程模型；信息流产生迭代；目标架构开发具有相互依存性。

6.4 关键术语 ▲难点

术语含义
ABBArchitecture Building Block，架构构建块
SBBSolution Building Block，解决方案构建块（E 阶段基于 ABB 定义）
过渡架构增量方法下交付连续业务价值的中间架构
基线架构 → 目标架构差距分析是 E/F 阶段核心
需求管理居中需求管理和利益相关者参与居于架构开发的中心

6.5 架构能力与 ACMM 成熟度

架构能力（EA 能力） = 开发、运用、维持企业架构并运用该架构治理变革的能力；通过组织结构、角色、职责、技能、流程落实。

ACMM 成熟度模型 6 阶段 ★高频：

  1. 独立建设（信息孤岛、重复建设）
  2. 项目化建设
  3. 企业级架构定义
  4. 企业级架构治理
  5. 中台化转型
  6. 数字化转型
典型企业当前所处：第二向第三阶段过渡。

6.6 业务架构设计（阶段 B）▲难点

传统职能型业务架构四类问题：需求散、建设散、系统散、响应慢（客户中心化偏弱是根源）。

业务架构设计五步：

  1. 基于波特价值链方法论拆解业务活动类型
  2. 对业务分解形成公司业务能力框架
  3. 基于业务能力识别 L3 级业务流程框架
  4. 基于端到端价值流方法构建主流程（含数据 CRUD/数据流/断点/指标分析）
  5. 沉淀共性组件，打造企业级组件复用（为业务中台奠基）
业务流程梳理到4 级（业务能力/活动级）。发展阶段：信息化=职能架构（以产品为中心）→ 数字化=赋能架构（以客户为中心）。

6.7 数据架构设计（阶段 C-数据）▲难点

5 条原则（必背）：

原则说明
数据完整性准确与完整，建立企业级数据模型
数据可用性任何时间/地点/服务连续可用
数据一致性不同系统同类数据取值一致
数据唯一性保证数据源唯一，减少冗余
可信数据源关键数据有权威可信源

数据资产三层（农场-工场-市场模型）：

层含义
数据农场数据资源治理（标准/质量/安全/认责）
数据工场数据授权运营（授权/汇聚/加工/经营）
数据市场数据价值释放（共享/开放/流通）
数据治理原则："谁生产、谁负责"。

6.8 应用架构设计（阶段 C-应用）

3 条原则：①完整性（全面覆盖业务）②集成性（纵横贯通，统一集成方案）③延续性（继承已有成果，弹性可扩展）。

应用功能形成三种方式：对现有系统功能分析 → 对业务组件直接使用 → 对业务组件打包组合（参数化）。沉淀共性应用组件是中台化基础。

6.9 技术架构设计（阶段 D）

目标：①实现系统技术构建 ②建立先进基础设施 ③实现基于技术平台的集成。

云服务四层：IaaS / PaaS / DaaS / IOTS（物联网服务）。

公用 IT 资产整合重用优势：资产利用率越高 IT 边际成本越低；显著降低应用推出时间；初期投入高，后续边际成本越来越低。

6.10 架构治理与落地 ★高频

业界案例（背书）：农行"薄前台、厚中台、强后台"；国家电网"多场景、微应用、一系统"；建行"一套业务模型、一套数字化架构、一套管理流程"。

散而不统 → 统一架构统一管控：立足顶层设计解决散的问题，统一架构、统一管控才是根本。

四统一：统一业务视图、统一数据视图、统一应用视图、统一平台视图（对应 BA/DA/AA/TA）。

架构落地"三化协同"：体系化、架构化、数字化；企业架构"建、管、用"。

管架构金句：无架构不制度、无架构不流程、无架构不数字化。

定位：基于社乡统一认证体系的 API 请求封装；卡片开发者一次性配置后，后续所有接口调用自动完成加密、签名、认证。

SDK 自动完成的处理（开发者无需关心）：

步骤机制
请求体加密AES-CBC 加密（data + orgId 合并）
认证令牌生成SHA-256 签名 + RSA 公钥加密
请求头注入X-DL-APP-ID、X-DL-WIDGET-ID、X-DL-Product-ID、Authorization、traceId
响应解密code === 0 时自动 AES 解密 res.data
认证过期code === 40105 时清除认证信息并触发回调

二、必背公式

CV 算力成本 ★高频▲难点

每路算力成本 ≈ 硬件成本 / 总路数   （当 路数 < QPS × 抽帧频率）
原则：无论一路加载多少算法，抽帧永远按最小抽帧频率一次抽帧。
省节点资源冗余 20% 作为安全区。

CV 标品审核阈值 ★高频

大目标：Precision > 92%，Recall > 80%
小目标：Recall > 70%，Precision > 80%
星河零代码 QPS > 80（普遍 > 100）；省自研 YOLO QPS > 50（最低不低于 50）
标品上架审核 6-8 周；接口审批 1 周左右

JSSDK 加密链 ★高频▲难点

请求体：AES-CBC（data + orgId）
认证令牌：SHA-256 签名 + RSA 公钥加密
成功解密：code === 0；认证过期：code === 40105
RSA 密钥：2048 位（PKCS#8 Base64）

认证时效（数字生活）

tokenCode：2 分钟一次有效；用户授权：30 天；数据订阅推送失败 3 次不再重试

TTT 记忆与语速 ★高频

分块 7±2；重复 6X；首要 1；最近 Z
最佳语速 120-200 字/分；停顿 10/6/3/2 秒

四、易混淆辨析

1. "CV 二开边界"

  • CV 二开 ≠ 拉流 + 省侧自建 AI，必须基于标训推平台 / 视联网标准能力。
  • 比对类算法：零代码和高代码均不支持（建议先检测后标准比对串联）。
  • 端侧视频流：零代码和高代码均需芯片适配。

2. "ThingsMQ 中间件是什么"

  • 融合应用的数采中间件是 ThingMQ，不是 Kafka / RabbitMQ。
  • MQTT 的 client_id 必须与设备编码一致；大屏撒点设备编码必须一致才能打通。

3. "TOGAF ADM 是顺序流程吗"

  • 不是。ADM 是逻辑方法，不应被视为顺序流程模型；信息流产生迭代；目标架构相互依存；需求管理居于中心。

4. "企业当前架构成熟度"

  • ACMM 6 阶段中，典型企业处于第二→第三阶段过渡（项目化建设 → 企业级架构定义）。

5. "实战考核是选择判断吗"

  • 不是。是《基层治理大屏设计》100 分纯实操：6 张卡片真实开发 + 真实调接口 + 审核通过 + 大屏编排 + 研发云全流程。
  • 大屏编排 35 分（最高单项）；研发云管理仅 5 分但必做。

6. "量子密信加密方式"

  • 消息一次一密、音视频全帧加密；基于国密标准 + 量子密钥；UI Kit 简化 80% 编码；Kit 与 Lib 可组合使用。
